# ベースライン負荷プロファイルシミュレーション

このシナリオでは「現在の構造とルールのもとで、どの役職がどれだけ負荷を抱えているのか？」という最小限の問いに答えることを目的としています。

## シナリオ概要
- **階層構造**: `CXO -> Director -> Manager -> Player`（CXO 1 名、Director 2 名、Manager 4 名、Player 8 名）。
- **隣接行列**: `baseline_load_profile.py` 内に、タスクの下流移動（実行）と上流報告（レポート）の両方を含めて定義し、流れやエスカレーションが明示されるようにしています。
- **タスク生成**: シミュレーションの各ステップで CXO ノードが 1〜3 件のタスクを発生させます。タスクの重みは  
  `w = difficulty * 1.2 + stakeholders * 0.8 + coordination * 0.5` で算出します。
- **フロー**:
  1. CXO から Director、Director から Manager、Manager から Player へランダムにタスクを引き渡す。
  2. Player は処理を完了後、即座に担当 Manager へ報告する。
  3. Manager は 95% を自律的に解決し、5% のみを担当 Director へエスカレーションする。
- **負荷計算**: タスクを渡すたびに受け手は重み `w` を、渡し手は調整コストとして `0.3w` を加算する。
- **出力**:
  - 各ノードに蓄積した総負荷。
  - CXO / Director / Manager / Player ごとの平均・分散。

## 実行方法
リポジトリ直下で以下を実行します。

```bash
python3 newSimulations/baseline_load_profile/baseline_load_profile.py
```

`random.seed(42)` で決定的になっており、標準ライブラリのみを使用するため追加の依存関係は不要です。

## 結果の読み方
出力には各ノードの負荷と役職別サマリが表示されます。合計値が大きいほどタスクを多く受け取っているか、頻繁に調整・エスカレーションしていることを意味します。同じ役職内のバラツキ（分散）を見ることで、特定メンバーに偏った負荷も検知できます。

`TIME_STEPS`、`TASKS_PER_CXO`、`ESCALATION_RATE` などを調整すれば、構造やルールを変えずに別のベースライン仮説を検証できます。
